# Gaming Track

Welcome to the **Gaming Track**.

This track is about learning how to **build a real video game**.

You will use **Godot** to make small playable games step by step. Along the way, you will learn how movement, collisions, rules, scoring, scenes, and game logic work.

If you like games, game ideas, animation, characters, levels, or figuring out how game rules work, this track is probably for you.

---

## What you will do in this track

In this track, you will:
- learn the basic parts of a game project
- make things move on the screen
- react to keyboard input
- use collisions, timers, and variables
- build small playable projects
- finish a game someone else can actually play
- show your work to a teacher or mentor

You do **not** need to make the next Minecraft.
You do need to make something real, playable, and understandable.

---

## What counts as progress?

Progress in this track should happen in **small steps**.

You should not wait until the end to prove you learned something. Each level should give you something playable, even if it is small.

There are three levels:
- **Bronze** - Make a simple playable scene with movement and input.
- **Silver** - Add rules, goals, or hazards so the game has a real challenge.
- **Gold** - Build a more complete game that someone else can test and understand.

Each level has:
- ideas to learn
- things to practice
- things to build
- a place for sign-off

---

## Why Godot?

This track uses **Godot**.

Godot is a game engine. A game engine is a tool that helps you build games without having to create every single system from scratch.

We are using Godot because:
- it is free
- it is beginner-friendly
- it works well for small 2D games
- it teaches real game development ideas

That means the goal is not just to click around randomly until something sort of moves. The goal is to understand how the game is put together.

---

## Setup and good habits

Some parts of this track may need help from a parent, teacher, or mentor, especially:
- installing Godot
- opening projects and finding files
- exporting a game for someone else to play
- saving projects in a clear folder structure

That is normal.

Good habits for this track:
- save your project often
- test your game every time you change something important
- keep old versions if you try something risky
- name scenes and files clearly
- when something breaks, change one thing at a time
- ask for help before you stay stuck for a thousand years

## Bronze Level

### Goal
Make a simple playable scene where the player can control something on the screen.

### Learn
At Bronze level, you should understand these ideas:
- what a **scene** is
- what a **node** is
- how keyboard input works
- how a sprite or object moves
- how to run and test a game

### Practice
Try to do these things:
- create a new Godot project
- add a player object or sprite
- move the player with the keyboard
- add a background or walls
- save your scene and run the game

### Bronze checkpoints
Complete these before asking for Bronze sign-off:
- [ ] I created a Godot project.
- [ ] I made an object move with keyboard input.
- [ ] I can explain what the player object is.
- [ ] I saved my project in a clear folder.
- [ ] I tested the game and fixed at least one problem.
- [ ] I can show a teacher or mentor how to run it.

### Bronze mini-project ideas
Pick at least **one**:
- move a character around a room
- move a spaceship around the screen
- make a simple maze with walls
- build a little scene where a character can walk around and explore

## Silver Level

### Goal
Turn your project into an actual game with rules, hazards, or goals.

### Learn
At Silver level, you should practice these ideas:
- what a collision is
- how to detect when two things touch
- how variables can store score, health, or time
- how to reset or restart a game
- how to make the player win or lose

### Practice
Try to do these things:
- add something the player should avoid or collect
- add a score, timer, or health value
- make something happen when the player wins or loses
- restart the game after failure or success

### Silver checkpoints
- [ ] I added a rule or goal to my game.
- [ ] I used collisions, overlap, or some other way to detect events.
- [ ] I used a variable such as score, health, or time.
- [ ] My game can be won, lost, or restarted.
- [ ] I tested the game more than once and fixed problems.
- [ ] I can explain what at least one important script does.

### Silver mini-project ideas
Pick at least **one**:
- dodge falling objects
- collect coins before time runs out
- survive a simple enemy or hazard
- escape a maze
- click or tap the right targets before time runs out

## Gold Level

### Goal
Build a stronger game that feels complete enough for another person to play and understand.

### Learn
At Gold level, the big idea is this:

> A good small game still needs clear rules, a real goal, and enough polish that another person can play it without you standing there explaining every second of it.

Your project does not need to be huge. It does need to feel like a real game, not just a test scene.

### Build
Create your own game using one of these ideas:
- a dodge game
- a maze game
- a platformer room or two
- a top-down collect-and-escape game
- a simple shooter
- a small puzzle game
- your own game idea

Your Gold project should include:
- a player-controlled character or object
- a goal, challenge, or objective
- a win condition or lose condition
- at least one script you understand and can explain
- enough polish that someone else can test it

Better version:
- add sound or music
- add a title screen or restart screen
- improve art, layout, or feedback
- organize your scenes and scripts clearly

### Gold checkpoints
- [ ] I built a game with a clear goal.
- [ ] My game has a win condition, lose condition, or both.
- [ ] My game uses at least one script that I can explain.
- [ ] My game works well enough for another person to test.
- [ ] I fixed bugs and improved the project after testing it.
- [ ] I saved my project clearly and kept my files organized.
- [ ] I showed the game to a teacher or mentor.

### Gold project examples
- Dodge enemies for 30 seconds to win.
- Collect 10 objects before time runs out.
- Escape a maze while avoiding hazards.
- Reach the end of a platforming level.
- Defeat a simple enemy or survive waves.
- Build your own small game idea with clear rules.

## If you get stuck

If your game is not working, try this:
- check one scene at a time
- check one script at a time
- test whether input is working first
- test whether collisions are working next
- print values or use simple messages to see what is happening
- ask someone else to play it and describe what broke

Sometimes the bug is not mysterious. Sometimes one box is unchecked or one object is named wrong. Game development is rude like that.
